什么是对象

在JAVA中，一切皆为对象。对象是类的实例，给类的所有属性赋值。

对象分为类对象和实例对象，实例对象很好理解。而类对象是Class类的一个实例，用来描述某个类或某个接口的定义。Class类的构造方法是私有的，这意味着类对象是不能被new出来的，而是由JVM在加载类的时候通过ClassLoader的defineClass方法自动创建的（这应该也意味着类对象都是单例的）。如果想获得一个类的实例，可以有两种方法：

对象的创建

在JAVA中，对象的创建有3种方式：

  1. 通过new关键字创建一个对象
  2. 通过Class对象的newInstance方法采用反射的方式创建一个对象
  3. 通过clone方法复制一个对象

创建对象时，会对对象的成员属性进行初始化，初始化语句的优先级顺序是:

  1. 属性的默认值
  2. 父类的静态语句块
  3. 子类的静态语句块
  4. 父类的无参构造方法
  5. 子类的构造方法

关于创建对象的一些最佳实践

使用静态工厂方法代替构造器

即通过暴露一些类的静态方法给客户端，在这些静态方法里帮助用户创建对象，而不是让用户直接通过new的方式。

这样做的第一个好处是，构造器其实是没有名字的，只能通过构造方法的签名（参数的类型、顺序）来区分不同的构造方法，这样用户在使用new的方式创建对象时可能会比较迷惑，不确定下一个该传什么参数（不过现在的一些智能IDE都会有提示的）。而通过静态工厂方法，可以通过给方法取合适的名字来屏蔽那些细节。

其实我们在工作中也有用到这个方法，比如在封装RPC服务的调用结果时，会提供一些调用成功或者失败的静态方法直接返回封装后的调用结果。

第二个好处是可以复用一些已经创建好的实例，避免重复创建对象。例如JDK中的Boolean.valueOf(boolean)、BigInteger.valueOf(long)方法等。

第三个好处是静态工厂方法可以返回原返回类型的子类的对象。这个有什么用呢？在一些基于接口的框架中，可能不方便返回对象的真正类型，例如JDK中的Collections集合工具，封装了一些返回不可变集合、同步集合的工具方法，其实在内部封装了很多不可变集合、同步集合的新的类型，但是客户端是不用感知的，而且将来如果替换了这些类型，也不会影响到客户端（这也要求客户端要采用接口来引用被返回的对象，当然这也是一种良好的习惯）。

在工作中，也会用到很多基于这种思想的框架或者库，比如SPI机制（JDBC的DriverManager.getConnection就是返回Connection服务接口的一个具体实现）。

因此，创建自定义类时，在提供公有的构造器之前，先想想使用静态工厂方法是否能带来上述好处。

当构造器参数过多时采用Builder模式

当类的构造器需要传入很多参数时，不管是重叠构造器模式，还是静态工厂方法，都会需要设置很多个不同的构造器或者静态方法。

JAVA beans方式（一个无参构造器+多个setter方法）也会导致状态的不一致，因为可能会忘了调用某一个setter方法。

这时可以考虑使用Builder模式来创建一个对象，代码易于阅读，也可以对参数进行校验，而且也可以保证bean状态的一致性。

Builder模式其实是模拟了像Python等语言中的具名可选参数。

Builder模式的缺点就是可能会使代码更加冗长，想要创建一个对象必须先创建它的builder对象，在一些对性能比较敏感的场景下可能会造成问题。

hashcode和equals方法

equals方法用来判断两个对象是否相等。相等分为”物理”相等和”逻辑”相等，物理相等是指两个对象指向同一个实例，即指向同一块内存区域。该方法的默认实现就是判断两个对象是否是”物理”相等的。如果自定义的类有判断”逻辑”相等的需求，就需要重写equals方法了。

重写equals方法需要遵循JDK规范中的通用约定，即要实现”等价关系”，包含以下几点：

  1. 自反性：对于任何非null的值x，x.equals(x)必须返回true。
  2. 对称性：对于任何非null的值x和y，当且仅当x.equals(y)返回true时，y.equals(x)也必须返回true。
  3. 传递性：对于任何非null的值x、y和z，如果x.equals(y)返回true，且y.equals(z)也返回true，则x.equals(z)也必须返回true。
  4. 一致性：对于任何非null的值x和y，只要x和y的对象信息没有被修改，那么x.equals(y)必须始终返回相同的结果。
  5. 非空性：对于任何非null的值x，x.equals(null)必须返回false。

结合上述要求，重写equals方法的一个最佳实践是：

  1. 使用==检查是否是同一个对象的引用。
  2. 使用instanceOf检查是否是正确的类型。
  3. 将参数转换为正确的类型。
  4. 对于每个”关键”域，检查是否相等。所谓的关键域，是指那些会影响对象逻辑相等的属性。最好不要包含冗余域，即那些可以通过关键域计算出来的域。

hashcode方法用来返回一个对象的散列值，本质上就是一个散列函数。JDK规范中对于该方法的实现约定是：

  1. 在应用的执行期间，只要对象的信息没有被修改，那么多次调用hashcode方法必须返回同一个整数值。
  2. 如果两个对象通过equals方法比较是相等的，那么它们的hashcode方法也必须返回相同的整数值。
  3. 如果两个对象通过equals方法比较是不相等的，那么就不要求它们的hashcode必须返回相同的整数值。

hashcode方法的默认实现是native的，它会返回对象在内存中的地址的整数值。对于equals和hashcode方法的默认实现，可以发现是符合上述要求的。

自定义类时，如果重写了equals方法时，必须也要重写hashcode方法，不然在和所有基于散列的集合（HashMap、HashSet、HashTable）一起使用时，可能会出现问题。比如自定义类的两个通过equals判定为相等的实例x和y，先以x为key放到散列集合中，再以y为key获取对应的value时，获取到的会是null，这是因为它们的散列值不同，会被放到不同的散列桶中。

那么具体要如何重写hashcode方法呢？一个好的散列函数会倾向于为不相等的对象生成不同的散列码。下面是一个简单通用的实现方法：

  1. 取一个非零的常量，比如17，作为变量result的值。
  2. 对于每个关键域，分别进行一下步骤：
    a. 计算int类型的散列码code。
     a.1 如果是boolean类型，计算`(f ? 1 : 0)`
 a.2 如果是byte、char、short或int类型，计算`(int)f`
 a.3 如果是long类型，计算`(int)(f ^ (f >>> 32))`
 a.4 如果是float类型，计算`Float.floatToIntBits(f)`
 a.5 如果是double类型，计算`Double.doubleToLongBits(f)`得到long类型的值，然后继续计算`(int)(f ^ (f >>> 32))`
 a.6 如果是对象引用，则通过递归的方式计算该对象的hashcode
 a.7 如果是数组，则按照递归的方式分别计算每个元素的值，然后再按照2.b的方法得到一个散列值。或者使用JDK1.5中增加的`Arrays.hashCode`方法
    b. 按照公式迭代计算result：result = 31 * result + c

上述2.b中的公式中之所以用31这个数字，是因为它是一个奇素数。如果乘数是偶数的话并且产生溢出的话，信息就会丢失（因为与2相乘等价于左移运算）；使用素数则是因为习惯使用素数计算散列结果。另外31还有个重要的特性，就是可以用移位运算和减法运算代替乘法（i * 31 == i << 5 - i），VM可以自动完成这种优化。

clone方法

clone方法用于创建一个对象的拷贝，它提供了一种不需要构造器就可以创建对象的方法。该方法有如下约定：

  1. x.clone() != x
  2. x.clone().getClass() == x.getClass()
  3. x.clone().equals(x)

但第2、3点都不是强制要求。

clone方法是protected的，因此它的子类实现时要改为public。此外想要实现clone方法的子类还必须实现Cloneable接口，表明该类拥有克隆对象的能力。如果没有实现Cloneable接口，调用clone方法会报CloneNotSupportedException异常。

clone方法虽然看上去很强大，很美好，但实际上用好却不容易。如果一个类只有基本类型的域或者指向不可变对象的引用域，那么按照如下方法重写即可:

但是如果一个类的域是指向可变对象的话，采用上述方法，新对象的域会指向被克隆的域所指向的对象，两者之间相互影响，就可产生不可预料的结果。因此对于这种情况，要采取”深度拷贝“的方法。可以有两种方法来实现：

  1. 递归调用每个域的clone方法
  2. 先调用super.clone()，然后把新对象的每个域设为空白状态，再通过调用高层方法来设置每个域的状态。比如拷贝一个散列表，先将每个散列桶置为空，然后使用put(key, value)方法将被克隆对象中的每个键值对放到新的散列桶中。

我们也可以使用其它方法来替代clone方法:

  1. 拷贝构造器：提供一个参数类型为类本身的构造器
  2. 拷贝工厂：提供一个类似于拷贝构造器的静态工厂方法

toString方法

该方法用来返回类的一个相对有意义的名称，默认实现是类名+hashcode的16进制表示。JDK规范推荐子类最好都能重写该方法。因为当一个对象通过prinln，字符串联操作符（+）或者调试器打印时，都会默认调用它的toString方法，如果返回的名称是有意义、便于人阅读的，将会有利于代码的调试。

在重写toString方法时，最好能够返回该对象中所有有意义的字段信息。

finalize方法

当一个对象不再可达，被垃圾收集器回收时，就会执行该对象的finalize方法。

但是执行finalize方法的时间是不确定的，因此不能像C++里的析构器那样用来做资源回收工作。此外，JDK规范里也没法保证finalize方法被哪一个线程执行，有可能执行finalize方法的线程优先级非常低。即使是System.gc和System.runFinalization方法也不能保证finalize方法一定会被执行。

因此还是要尽量避免使用finalize方法。
